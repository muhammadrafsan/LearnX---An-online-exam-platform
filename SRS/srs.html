1. Introduction


1.1 Purpose

The primary purpose to develop this system is to optimize the online examination system for any organization. An exam evaluator can create an exam-cohort and add users to it, where the users can participate in online examination in the form of MCQ and micro-viva. The system will automatically evaluate all the answers and generate a marksheet which is downloadable

1.2 Document Conventions

The font used in this document is Arial having size 16. The text is made bolt for the highlighting purpose so that it can be easily differentiated. In this document, every functionality is equally important and every requirement has its own priority.

1.3 Intended Audience and Reading Suggestions

The document is meant for the developers, testers, designers and users which can be anyone willing to use the system. This document will serve as a reference document for every team which is partaking in making the project. They will coordinate every activity that take place in the software engineering process and will be guided by Dr. Nabeel Mohammed.

1.4 Product Scope

Online Examination System enables the users to have a robust exam taking facilities and features at its disposal. It resolves the typical issues of manually conducting examinations where both teachers and students have to attend. The objective of this application is to allow users to create and participate in examinations at their fingertips sitting anywhere in the world. This kind of system is specially designed for educational institutions to help in taking examinations smoothly.

2. Overall Description


2.1 Product Perspective

Online examination system is created to ease the complications of conducting and partaking in an online examination. Users can log into their account and create an exam-cohort where they can add other users to conduct an online examination. The system is self-contained.

2.2 Product Functions

The system’s functions are super user-friendly so that it is easy for the users to operate. The major functions of the software are provided below.

  • All users can create an account and login using google account.
  • Those who created an exam-cohort will have editing abilities i.e. create, edit, delete assessments, add other users to the cohort.
  • Other users can sit for the exam and also create an exam-cohort if they want.
  • The system admin can delete any users at any moment.
    •
2.3 User Classes and Characteristics

All users can access the software Online Examination System from two different platforms: Web app and Android app. Android users can only sit for an exam, not create one. There are two different types of actors, they are system admin and users. System admins has the highest power i.e. they can add or delete account. Users can log into the system. Users such as the evaluators can create an exam-cohort, add students to it and then create, edit assessments. Lastly they can download a spreadsheet of marks. Users such as students can sit for an exam using both the web app and mobile app.

2.4 Operating Environment
  • Platform supported: Web and Android
  • Web server: Any operation system with Django 4.1 or newer installed
  • Android app: Any system with a functional web browser.
  • Database configuration: SQLite
    •
2.5 Design and Implementation Constraints

We have a web app and android app, but no IOS app for the system. Users can only take the exam through android app. Users cannot use other features inside the mobile app.

2.6 User Documentation

The product will come with a user manual which will explain how to use the system such as how to create an exam-cohort, how to add students, how to create an assessment, etc.

2.7 Assumptions and Dependencies

Users must have a stable and working internet connection. Web users must have a working browser to use the product. Mobile users must have android version 6+ installed.



3. External Interface Requirements


3.1 User Interfaces

TThe users will start off at the login page where they will find a button of google login. Users can log in using a valid google account. After they are logged in, they can create an exam cohort where they can add other users (students). Inside the cohort the user (evaluator) can create an assessment and add question in the form of MCQs and micro-viva which is editable. After setting an assessment, users (students) can sit for the exam. After due time of the exam, system will automatically calculate the marks and the evaluator will be able to download a marksheet showing the mark breakdown of every student. Evaluators can edit the questions, answers and marks.

3.2 Hardware Interfaces

Since the mobile application does not have designated hardware, it does not have any hardware interface. The hardware connection between the database server and application is managed by the underlying operating system on the mobile phone.

3.3 Software Interfaces

The system is self-contained and no data is supposed to share with the third party. The communication of the mobile application between the database consist of only reading the data, no modification is allowed. The communication between the web application and database consist of both reading and modifying.

3.4 Communications Interfaces

The communication between the different parts of the system is important since they depend on each other. However, in what way the communication is achieved is not important for the system and is therefore handled by the underlying operating system for the mobile application.



4. System Features


4.1 Log in using Google

Use Case: Login using Google
Actor(s): User
Purpose: The user can login to this system using their Google account, and then use this system.
Type: Primary

Typical course of events:
Actor Action System Response
1. User clicks the login from login page. 2. System redirects to the google login page
3. User chooses a google account 4. System will response by verifying the credentials through google API.
Alternative Course of Events

Section 3: If the user enter incorrect google credentials, the system will redirect to login page.

4.2 Create an exam-cohort

Use Case: Create an exam-cohort
Actor(s): User
Purpose: After logging into the system, the user can create an exam cohort by clicking the "create exam-cohort" button
Type: Primary

Typical course of events:
Actor Action System Response
1. User clicks “Create an exam-cohort” button. 2. The system provides the create an exam-cohort form.
3. User fills up the necessary information and clicks submit button. 4. System will show a successful message and the cohort will be added to the dashboard.
Alternative Course of Events

Section 2: The user might not been logged into his/her google account, the system will redirect to the login page

Section 4: The system will show an error if the form is not filled properly/if some information is missing.

4.3 Create assessments within a cohort

Use Case: Create assessments within a cohort
Actor(s): Evaluator
Purpose: The evaluator of an exam cohort can create assessments within that cohort.
Type: Primary

Typical course of events:
Actor Action System Response
1. This use case starts when the evaluator clicks on the "Add an assessment" button. 2. The system redirects to the assessment creation form.
3. Evaluator will fill up the form and click submit. 4. System will show a successful message and will add that assessment to that cohort's dashboard.
Alternative Course of Events

Section 4: The system will show an error if the form is not filled properly/if some information is missing.

4.4 Add users (students) to the cohort

Use Case: Add users (students) to the cohort
Actor(s): Evaluator
Purpose: The evaluator can add students to the exam cohort so that they can take the exam.
Type: Primary

Typical course of events:
Actor Action System Response
1. This use case starts when the evaluator clicks the "add student" button. 2. System redirects to the add student page.
3. The evaluator will enter the student's email address and then click submit button. 4. System will add the student to that cohort.
5. The system will add the exam-cohort to the dashboard of the student as well.
Alternative Course of Events

Section 4: There can be an error because the entered email might be invalid.

4.5 Edit assessments

Use Case: Edit assessments
Actor(s): Evaluator
Purpose: The evaluator can edit a created assessment.
Type: Primary

Typical course of events:
Actor Action System Response
1. The use case starts when the user clicks “Edit an assessment”. 2. The system provides the assessment page, which is editable
3. User does necessary changes to the assessment and saves it. 4. System will correct those mistakes and show the corrected assessment.
Alternative Course of Events
4.6 Add MCQs to the assessment

Use Case: Add MCQs to the assessment
Actor(s): Evaluator
Purpose: The evaluator of an exam cohort can add MCQ questions within that cohort and specify the correct answer.
Type: Primary

Typical course of events:
Actor Action System Response
1. This use case starts when the evaluator clicks "Add MCQs" 2. The system will provide a text box to write the questions and assign the mark.
3. Evaluator will write the question and specify the marks as well as the correct answer for that question and click save. 4. System will save that question and add it to the assessment.
Alternative Course of Events

Section 3: If the specified mark for an MCQ is more than the total mark of the assessment, then an error will be shown.

Section 4: If the evaluator clicks the save button without specifying the correct answer and mark, then it will show an error.

4.7 Add micro-viva to the assessment

Use Case: Add micro-viva to the assessment
Actor(s): Evaluator
Purpose: The evaluator of an exam cohort can add micro-viva questions within that cohort and specify the possible correct answers.
Type: Primary

Typical course of events:
Actor Action System Response
1. This use case starts when the evaluator clicks "Add micro-viva" 2. The system will provide an option to record an audio question using the microphone.
3. Evaluator will record the question and specify the marks as well as the possible correct answers for that question and click save. 4. System will save that question and add it to the assessment.
Alternative Course of Events

Section 3: If the specified mark for a micro-viva is more than the total mark of the assessment, then an error will be shown.

Section 3: The system will not be able to record the questions if the voice is not clear or the if the microphone doesn't work properly.

Section 4: If the evaluator clicks the save button without specifying the correct answer and mark, then it will show an error.

4.8 Download a marksheet

Use Case: 4.8 Download a marksheet
Actor(s): Evaluator
Purpose: The evaluator of an exam cohort can download the marksheet of the students after the exam. The marksheet will have the breakdowns for all marks.
Type: Primary

Typical course of events:
Actor Action System Response
1. This use case starts when the evaluator clicks "Download Marksheet" 2. The system will download that file
Alternative Course of Events

Section 2: The system might download the wrong marksheet because it might not been renamed in a systematic pattern.

4.9 Take the exam

Use Case: Take the exam
Actor(s): Students
Purpose: The students of an exam cohort can take the exam by answering the questions assigned in the assessment. It can be done through web and mobile application.
Type: Primary

Typical course of events:
Actor Action System Response
1. This use case starts when the student clicks "Start the exam" 2. The system will redirect the student to the exam window.
3. The student will answer the mcqs by ticking the correct option, and for micro-viva, he/she will answer using his/her voice. 5. System will save the answer and return the marks.
4. The student will click the submit button after the exam finishes. 6. System will also send these answers to the evaluator so that he/she can edit the marks if needed.
Alternative Course of Events

Section 5: If internet fails during the exam, the marks might be zero.

4.10 Log out

Use Case: Log out
Actor(s): Users
Purpose: The users can log out of his/her account using the logout button.
Type: Primary

Typical course of events:
Actor Action System Response
1. This use case starts when the user clicks the Logout button. 2. The system will redirect the user to the login page.
Alternative Course of Events


5. Other Nonfunctional Requirements


5.1 Performance Requirements
  • Both android app and web app is using the same backend which ensures good performance and synchronization.
  • The system is well optimized to tackle large volume of traffic at once.
  • System is available 24 by 7.
5.2 Safety Requirements
  • Errors will be minimized and an appropriate error message that guides the user from an error will be provided.
  • Validation of users input is highly essential.
5.3 Security Requirements
  • The system is provided a high level of security and integrity of the data by the system.
  • Only authorized personnel such as admin can gain access to the private data and only the user with a valid google account is allowed to create an account and view its user page.
5.4 Software Quality Attributes
  • The key software quality attributes are Availability, Reliability and Usability.
  • As the system is expected to be working 24/7. High availability is important.
  • A simple but quality user interface is developed to make it easy to understand.
  • The error message displayed is more descriptive and can be easily understood.
5.5 Business Rules
  • System shall be available for everyone.
  • All the users shall access the system using a valid google account.
