1. Introduction

The primary purpose of the Software Design Specification (SDS) document is to provide information about the design and implementations of our project "Online Examination System". This SDS document will provide a detailed description of the design components, high level class diagrams along with the description of each classes that are used in our system, database design and finally, the plan of our implementation.


2. Document Conventions

The font used in this document is Arial having size 16. The text is made bolt for the highlighting purpose so that it can be easily differentiated. In this document, every functionality is equally important and every requirement has its own priority.


3. Description of Design Components

This project will have a Web Application as well as a Mobile Application (Android).
From the web application, users can get access to the following features:

  • Login to the System using Google account
  • Create an exam cohort
  • Add students to the cohort
  • Create an assessment within that cohort
  • Inside each assessment, the evaluator can assign questions (MCQ or Micro-viva)
  • The evaluator can specify the time, mark and the correct answer for those questions
  • The students can participate in the assessments
  • After the exam, the evaluator can see the mark and edit them if required
  • The marksheet can be downloaded
  • The users can log out of the system

From the mobile application (Android), users can get access to the following features:

  • Login to the System using Google account
  • The students can participate in the assessments
  • The users can log out of the system

6. Database Design


6.1 ER Diagram

The ER Diagram is attached below:


Purpose
  • Examcohort: The Examcohort table is used to keep the records of all exam cohorts that are created in the system.
  • Assessment: The Assessment table is used to store all the information about the assessments that are created within a cohort.
  • Question: This table stores the information regarding each question that are posted in the assessment.
  • MCQ: The MCQ table stores the information regarding each MCQ that are posted in the assessment.
  • Micro-viva: The Micro-viva table stores the information regarding each viva that are posted in the assessment.
  • Student: This table keeps record of all the necessary information of the students that are added in the cohort.
  • Evaluator: The Evaluator table keeps record of all the necessary information of the evaluator, who created a cohort.
  • Spreadsheet: This table stores the marks of all the students participated in an assessment.


7. Implementation plan

  1. At first, we decided to gather the requirements for our project.
  2. Then, we'll analyses those requirements and make an SRS.
  3. We will design our project and produce an SDS.
  4. After that, we will divide the tasks between the group members.
  5. We set deadlines for specific tasks (Trello Board).
  6. Now, we will develop the system.
  7. We will then carry out unit testing on our developed project.
